Eloquent: Getting Started

Source: https://laravel.com/docs/12.x/eloquent

Introduction

Laravel includes Eloquent, an object-relational mapper (ORM) that makes it enjoyable to interact with your database. When using Eloquent, each database table has a corresponding "Model" that is used to interact with that table. In addition to retrieving records from the database table, Eloquent models allow you to insert, update, and delete records from the table as well.

Before getting started, be sure to configure a database connection in your application's `config/database.php` configuration file.

Generating Model Classes

To get started, let's create an Eloquent model. Models typically live in the `app\Models` directory and extend the `Illuminate\Database\Eloquent\Model` class. You may use the `make:model` Artisan command to generate a new model:

```shell
php artisan make:model Flight
```

If you would like to generate a database migration when you generate the model, you may use the `--migration` or `-m` option:

```shell
php artisan make:model Flight --migration
```

You may generate various other types of classes when generating a model, such as factories, seeders, policies, controllers, and form requests:

```shell
Generate a model and a migration, factory, seeder, and controller...
php artisan make:model Flight -mfsc

Shortcut to generate a model, migration, factory, seeder, policy, controller, and form requests...
php artisan make:model Flight --all
```

Eloquent Model Conventions

Models generated by the `make:model` command will be placed in the `app/Models` directory. Let's examine a basic model class:

```php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Model;

class Flight extends Model
{
    // ...
}
```

Table Names

By convention, the "snake case", plural name of the class will be used as the table name unless another name is explicitly specified. So, in this case, Eloquent will assume the `Flight` model stores records in the `flights` table.

If your model's corresponding database table does not fit this convention, you may manually specify the model's table name by defining a `table` property on the model:

```php
class Flight extends Model
{
    protected $table = 'my_flights';
}
```

Primary Keys

Eloquent will also assume that each model's corresponding database table has a primary key column named `id`. If necessary, you may define a protected `$primaryKey` property on your model to specify a different column:

```php
class Flight extends Model
{
    protected $primaryKey = 'flight_id';
}
```

In addition, Eloquent assumes that the primary key is an incrementing integer value. If you wish to use a non-incrementing or a non-numeric primary key you must define a public `$incrementing` property on your model that is set to `false`:

```php
class Flight extends Model
{
    public $incrementing = false;
}
```

UUID and ULID Keys

Instead of using auto-incrementing integers as your Eloquent model's primary keys, you may choose to use UUIDs instead. If you would like a model to use a UUID key instead of an auto-incrementing integer key, you may use the `Illuminate\Database\Eloquent\Concerns\HasUuids` trait on the model:

```php
use Illuminate\Database\Eloquent\Concerns\HasUuids;
use Illuminate\Database\Eloquent\Model;

class Article extends Model
{
    use HasUuids;

    // ...
}

$article = Article::create(['title' => 'Traveling to Europe']);

$article->id; // "018f2b5c-6a7f-7b12-9d6f-2f8a4e0c9c11"
```

Timestamps

By default, Eloquent expects `created_at` and `updated_at` columns to exist on your model's corresponding database table. If you do not want these columns to be automatically managed by Eloquent, define a `$timestamps` property on your model with a value of `false`:

```php
class Flight extends Model
{
    public $timestamps = false;
}
```

Retrieving Models

Once you have created a model and its associated database table, you are ready to start retrieving data from your database. You can think of each Eloquent model as a powerful query builder allowing you to fluently query the database table associated with the model. The model's `all` method will retrieve all of the records from the model's associated database table:

```php
use App\Models\Flight;

foreach (Flight::all() as $flight) {
    echo $flight->name;
}
```

Building Queries

Since each Eloquent model serves as a query builder, you may add additional constraints to queries and then invoke the `get` method to retrieve the results:

```php
$flights = Flight::where('active', 1)
    ->orderBy('name')
    ->limit(10)
    ->get();
```

Chunking Results

Your application may run out of memory if you attempt to load tens of thousands of Eloquent records via the `all` or `get` methods. Instead, the `chunk` method may be used to process large numbers of models more efficiently:

```php
use App\Models\Flight;
use Illuminate\Database\Eloquent\Collection;

Flight::chunk(200, function (Collection $flights) {
    foreach ($flights as $flight) {
        // ...
    }
});
```

Cursors

Similar to the `lazy` method, the `cursor` method may be used to significantly reduce your application's memory consumption when iterating through tens of thousands of Eloquent model records. Internally, the `cursor` method uses PHP generators:

```go
```

```php
use App\Models\Flight;

foreach (Flight::where('destination', 'Zurich')->cursor() as $flight) {
    // ...
}
```

Retrieving Single Models / Aggregates

In addition to retrieving all of the records matching a given query, you may also retrieve single records using the `find`, `first`, or `firstWhere` methods:

```php
use App\Models\Flight;

// Retrieve a model by its primary key...
$flight = Flight::find(1);

// Retrieve the first model matching the query constraints...
$flight = Flight::where('active', 1)->first();

// Alternative to retrieving the first model matching the query constraints...
$flight = Flight::firstWhere('active', 1);
```

Not Found Exceptions

Sometimes you may wish to throw an exception if a model is not found. The `findOrFail` and `firstOrFail` methods will retrieve the first result of the query; however, if no result is found, an `Illuminate\Database\Eloquent\ModelNotFoundException` will be thrown:

```php
$flight = Flight::findOrFail(1);

$flight = Flight::where('legs', '>', 3)->firstOrFail();
```

If the `ModelNotFoundException` is not caught, a 404 HTTP response is automatically sent back to the client.

Retrieving or Creating Models

The `firstOrCreate` method will attempt to locate a database record using the given column / value pairs. If the model cannot be found in the database, a record will be inserted:

```php
use App\Models\Flight;

// Retrieve flight by name or create it if it doesn't exist...
$flight = Flight::firstOrCreate([
    'name' => 'London to Paris'
]);

// Retrieve flight by name or create it with the name, delayed, and arrival_time attributes...
$flight = Flight::firstOrCreate(
    ['name' => 'London to Paris'],
    ['delayed' => 1, 'arrival_time' => '11:30']
);
```

Inserting and Updating Models

Inserts

To insert a new record into the database, you should instantiate a new model instance and set attributes on the model. Then, call the `save` method on the model instance:

```php
namespace App\Http\Controllers;

use App\Models\Flight;
use Illuminate\Http\RedirectResponse;
use Illuminate\Http\Request;

class FlightController extends Controller
{
    public function store(Request $request): RedirectResponse
    {
        // Validate the request...

        $flight = new Flight;

        $flight->name = $request->name;

        $flight->save();

        return redirect('/flights');
    }
}
```

Alternatively, you may use the `create` method to "save" a new model using a single PHP statement:

```php
use App\Models\Flight;

$flight = Flight::create([
    'name' => 'London to Paris',
]);
```

However, before using the `create` method, you will need to specify either a `fillable` or `guarded` property on your model class, since all Eloquent models are protected against mass assignment vulnerabilities by default.

Updates

The `save` method may also be used to update models that already exist in the database:

```php
use App\Models\Flight;

$flight = Flight::find(1);

$flight->name = 'Paris to London';

$flight->save();
```

Occasionally, you may need to update an existing model or create a new model if no matching model exists. The `updateOrCreate` method persists the model, so there's no need to manually call the `save` method:

```php
$flight = Flight::updateOrCreate(
    ['departure' => 'Oakland', 'destination' => 'San Diego'],
    ['price' => 99, 'discounted' => 1]
);
```

Mass Updates

Updates can also be performed against models that match a given query:

```php
Flight::where('active', 1)
    ->where('destination', 'San Diego')
    ->update(['delayed' => 1]);
```

Mass Assignment

A mass assignment vulnerability occurs when a user passes an unexpected HTTP request field and that field changes a column in your database that you did not expect. So, to get started, you should define which model attributes you want to make mass assignable using the `$fillable` property:

```php
class Flight extends Model
{
    protected $fillable = ['name'];
}
```

Once you have specified which attributes are mass assignable, you may use the `create` method to insert a new record:

```php
$flight = Flight::create(['name' => 'London to Paris']);
```

If you would like to make all of your attributes mass assignable, you may define your model's `$guarded` property as an empty array:

```php
protected $guarded = [];
```

Upserts

Eloquent's `upsert` method may be used to update or create records in a single, atomic operation:

```php
Flight::upsert([
    ['departure' => 'Oakland', 'destination' => 'San Diego', 'price' => 99],
    ['departure' => 'Chicago', 'destination' => 'New York', 'price' => 150]
], uniqueBy: ['departure', 'destination'], update: ['price']);
```

Deleting Models

To delete a model, you may call the `delete` method on the model instance:

```php
use App\Models\Flight;

$flight = Flight::find(1);

$flight->delete();
```

Deleting an Existing Model by its Primary Key

If you know the primary key of the model, you may delete the model without explicitly retrieving it by calling the `destroy` method:

```php
Flight::destroy(1);

Flight::destroy(1, 2, 3);

Flight::destroy([1, 2, 3]);
```

Deleting Models Using Queries

You may build an Eloquent query to delete all models matching your query's criteria:

```php
$deleted = Flight::where('active', 0)->delete();
```

Soft Deleting

In addition to actually removing records from your database, Eloquent can also "soft delete" models. When models are soft deleted, they are not actually removed from your database. Instead, a `deleted_at` attribute is set on the model. To enable soft deletes for a model, add the `Illuminate\Database\Eloquent\SoftDeletes` trait:

```php
namespace App\Models;

use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\SoftDeletes;

class Flight extends Model
{
    use SoftDeletes;
}
```

To determine if a given model instance has been soft deleted, you may use the `trashed` method:

```php
if ($flight->trashed()) {
    // ...
}
```

Restoring Soft Deleted Models

To restore a soft deleted model, you may call the `restore` method on a model instance:

```php
$flight->restore();
```

Permanently Deleting Models

You may use the `forceDelete` method to permanently remove a soft deleted model from the database table:

```php
$flight->forceDelete();
```

Querying Soft Deleted Models

Soft deleted models will automatically be excluded from query results. However, you may force soft deleted models to be included in a query's results by calling the `withTrashed` method:

```php
use App\Models\Flight;

$flights = Flight::withTrashed()
    ->where('account_id', 1)
    ->get();
```

The `onlyTrashed` method will retrieve only soft deleted models:

```php
$flights = Flight::onlyTrashed()
    ->where('airline_id', 1)
    ->get();
```

Laravel: Eloquent Relationships — hasOne, hasMany, belongsTo

One to One / Has One

A one-to-one relationship is a basic database relationship where a parent model is associated with one child model. Example: a User model might have one Phone model.

class User extends Model
{
    public function phone(): HasOne
    {
        return $this->hasOne(Phone::class);
    }
}

Access it: $phone = User::find(1)->phone;

By default, Eloquent assumes the foreign key is the parent model name with an _id suffix (e.g. user_id). Override it:

return $this->hasOne(Phone::class, 'foreign_key');
return $this->hasOne(Phone::class, 'foreign_key', 'local_key'); // custom local key too

Define the inverse on the child model with belongsTo:

class Phone extends Model
{
    public function user(): BelongsTo
    {
        return $this->belongsTo(User::class);
    }
}

Access the parent: $user = $phone->user;

One to Many / Has Many

A one-to-many relationship defines where a single parent model relates to multiple child models. Example: a blog Post may have many Comment models.

class Post extends Model
{
    public function comments(): HasMany
    {
        return $this->hasMany(Comment::class);
    }
}

Access it: $comments = Post::find(1)->comments;

Query with constraints:

$comment = Post::find(1)->comments()->where('title', 'foo')->first();

Customize foreign/local keys the same way as hasOne: $this->hasMany(Comment::class, 'foreign_key', 'local_key').

Preventing N+1 query problems with chaperone(): when eager loading and accessing parent models from children, chain chaperone() to avoid repeated queries.

class Post extends Model
{
    public function comments(): HasMany
    {
        return $this->hasMany(Comment::class)->chaperone();
    }
}

Or at query time:

$posts = Post::with(['comments' => fn ($comments) => $comments->chaperone()])->get();

One to Many (Inverse) / Belongs To

Define the inverse of a hasMany relationship on the child model. Example: a Comment belongs to one Post.

class Comment extends Model
{
    public function post(): BelongsTo
    {
        return $this->belongsTo(Post::class);
    }
}

Access it: $comment->post->title

Default Models

Return a default model when the relationship is null (Null Object pattern), optionally with default attributes or a closure:

public function user(): BelongsTo
{
    return $this->belongsTo(User::class)->withDefault(['name' => 'Guest Author']);
}

Querying Belongs To Relationships

// Manual where clause
$posts = Post::where('user_id', $user->id)->get();

// Using the whereBelongsTo helper
$posts = Post::whereBelongsTo($user)->get();

// Specify the relationship name explicitly
$posts = Post::whereBelongsTo($user, 'author')->get();

Many to Many Relationships

Many-to-many relationships are slightly more complex than hasOne and hasMany, requiring an intermediate join table. For example, a User can belong to many Role models, and a Role can belong to many User models, so a roles table alone can't express this without a pivot table linking the two, conventionally named role_user with the two related model names in alphabetical order:

class User extends Model
{
    public function roles(): BelongsToMany
    {
        return $this->belongsToMany(Role::class);
    }
}

Access it like any other relationship: $roles = User::find(1)->roles;

Because the join table itself has no corresponding Eloquent model by default, additional columns on the pivot table are accessed through a pivot attribute automatically present on each retrieved model:

foreach ($user->roles as $role) {
    echo $role->pivot->created_at;
}

Retrieving Intermediate Table Columns

To specify which pivot columns should be included beyond the default id fields, use withPivot:

return $this->belongsToMany(Role::class)->withPivot('active', 'created_by');

Filtering Queries via Intermediate Table Columns

wherePivot, wherePivotIn, and related methods let you filter the relationship query using columns on the pivot table itself:

return $this->belongsToMany(Role::class)->wherePivot('approved', 1);

Has One of Many / Has Many of Many

Sometimes a model has many related models but you're only interested in the "latest" or "oldest" one — for example, a Customer might have many Order models but you want quick access to the most recent one:

public function latestOrder(): HasOne
{
    return $this->hasOne(Order::class)->latestOfMany();
}

public function oldestOrder(): HasOne
{
    return $this->hasOne(Order::class)->oldestOfMany();
}

Polymorphic Relationships

A polymorphic relationship allows the child model to belong to more than one type of model using a single association. For example, both a Post and a Video might be commentable, and rather than a separate comments table per model, a single Comment model can belong to either:

class Comment extends Model
{
    public function commentable(): MorphTo
    {
        return $this->morphTo();
    }
}

class Post extends Model
{
    public function comments(): MorphMany
    {
        return $this->morphMany(Comment::class, 'commentable');
    }
}

class Video extends Model
{
    public function comments(): MorphMany
    {
        return $this->morphMany(Comment::class, 'commentable');
    }
}

The comments table stores both commentable_id and commentable_type columns, letting a single table serve as comments for any number of different "commentable" model types without a comments_for_posts and comments_for_videos table pair.

Eager Loading

Eager loading alleviates the N+1 query problem, where accessing a relationship on each of N models triggers N additional queries. The with method loads a relationship up front, in a single additional query per relationship, rather than lazily when it's first accessed on each model:

$books = Book::with('author')->get();

Eager load multiple relationships, or nested relationships using dot notation:

$books = Book::with(['author', 'publisher'])->get();
$books = Book::with('author.contacts')->get();

Constraining eager loads with a closure lets you filter or sort the loaded relationship itself:

$users = User::with(['posts' => function (Builder $query) {
    $query->where('title', 'like', '%code%');
}])->get();


Laravel: Database Migrations

Source: https://laravel.com/docs/12.x/migrations

Introduction

Migrations are like version control for your database, allowing your team to define and share the application's database schema definition. Migrations are typically paired with Laravel's schema builder to easily build your application's database schema, and are stored as individual PHP files in the database/migrations directory.

Generating Migrations

php artisan make:migration create_flights_table

The migration will be placed in the database/migrations directory, and each migration filename contains a timestamp that allows Laravel to determine the order of the migrations. If you'd like to specify the table name, or whether the migration will be creating a new table, you may use the --table and --create options:

php artisan make:migration create_flights_table --create=flights
php artisan make:migration add_paid_to_flights_table --table=flights

Migration Structure

A migration class contains two methods: up, run when applying the migration (typically creating tables, columns, or indexes), and down, run when rolling back the migration, which should reverse the operations performed by up:

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('flights', function (Blueprint $table) {
            $table->id();
            $table->string('name');
            $table->string('airline');
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('flights');
    }
};

Running Migrations

php artisan migrate

To see which migrations have run so far:

php artisan migrate:status

Rolling Back Migrations

To roll back the most recent "batch" of migrations, which may include multiple migration files:

php artisan migrate:rollback

Roll back a limited number of migrations by providing a step option:

php artisan migrate:rollback --step=5

The migrate:reset command rolls back all of your application's migrations entirely:

php artisan migrate:reset

Roll Back and Migrate in a Single Command

The migrate:refresh command rolls back all of your migrations and then runs the migrate command again, effectively re-creating your entire database:

php artisan migrate:refresh

Tables

Creating Tables

Schema::create('users', function (Blueprint $table) {
    $table->id();
    $table->string('name');
    $table->string('email');
    $table->timestamps();
});

Checking for Table / Column Existence

if (Schema::hasTable('users')) {
    // ...
}

if (Schema::hasColumn('users', 'email')) {
    // ...
}

Updating Tables

The table method on the Schema facade is used to update existing tables, accepting the table's name and a closure receiving a Blueprint instance for adding columns or indexes:

Schema::table('users', function (Blueprint $table) {
    $table->integer('votes');
});

Renaming / Dropping Tables

Schema::rename($from, $to);
Schema::drop('users');
Schema::dropIfExists('users');

Columns

Common Column Types

$table->string('name', 100);
$table->text('description');
$table->integer('votes');
$table->boolean('confirmed');
$table->date('created_at');
$table->timestamp('added_at');
$table->json('options');
$table->uuid('id');

Column Modifiers

Modifiers can be chained onto a column definition to adjust its behavior:

$table->string('email')->nullable();
$table->integer('votes')->default(0);
$table->string('email')->unique();
$table->timestamp('added_at')->useCurrent();

Modifying Columns

The change method allows you to modify the type and attributes of existing columns:

Schema::table('users', function (Blueprint $table) {
    $table->string('name', 50)->change();
});

Dropping Columns

Schema::table('users', function (Blueprint $table) {
    $table->dropColumn('votes');
});

// Drop multiple columns
Schema::table('users', function (Blueprint $table) {
    $table->dropColumn(['votes', 'avatar', 'location']);
});

Indexes

Creating Indexes

$table->string('email')->unique();

// Or as a separate statement, useful for composite indexes
$table->index(['account_id', 'created_at']);

Foreign Key Constraints

Laravel also provides support for creating foreign key constraints, used to force referential integrity at the database level. The foreignId method is a convenient shorthand for adding a common unsignedBigInteger foreign key column:

Schema::table('posts', function (Blueprint $table) {
    $table->foreignId('user_id')->constrained();
});

You may specify the desired action for the "on delete" and "on update" properties of the constraint:

$table->foreignId('user_id')
    ->constrained()
    ->onUpdate('cascade')
    ->onDelete('cascade');

Dropping Foreign Keys

Schema::table('posts', function (Blueprint $table) {
    $table->dropForeign(['user_id']);
});


Laravel: Request Validation Basics

Laravel provides multiple approaches to validate incoming request data, with the validate method being the most common. The framework includes a wide variety of convenient validation rules and supports validating against database tables for uniqueness.

Quick Start: Validation Flow

Define routes:

Route::get('/post/create', [PostController::class, 'create']);
Route::post('/post', [PostController::class, 'store']);

Validate inside the controller:

public function store(Request $request): RedirectResponse
{
    $validated = $request->validate([
        'title' => ['required', 'unique:posts', 'max:255'],
        'body' => ['required'],
    ]);

    // The blog post is valid...
    return redirect('/posts');
}

Automatic Response Handling

For traditional HTTP requests, validation failure automatically redirects back to the previous URL with the errors flashed to the session. For XHR requests, a JSON response with a 422 status code is returned containing the validation errors. On success, code continues executing normally.

Stopping on First Failure

Use the bail rule to stop validating a field after its first failure:

$request->validate([
    'title' => ['bail', 'required', 'unique:posts', 'max:255'],
    'body' => ['required'],
]);

Nested Field Validation

Use dot notation for nested fields:

$request->validate([
    'author.name' => ['required'],
    'author.description' => ['required'],
]);

Displaying Validation Errors

The $errors variable is automatically available in all views via the ShareErrorsFromSession middleware:

@if ($errors->any())
    <div class="alert alert-danger">
        <ul>
            @foreach ($errors->all() as $error)
                <li>{{ $error }}</li>
            @endforeach
        </ul>
    </div>
@endif

Or per-field with the @error directive:

<input id="title" type="text" name="title" class="@error('title') is-invalid @enderror" />
@error('title')
    <div class="alert alert-danger">{{ $message }}</div>
@enderror

Repopulating Forms

Laravel automatically flashes request input on validation failure. Retrieve old input with $request->old('title') or, in Blade, {{ old('title') }}.

Optional Fields

Mark optional fields as nullable, since Laravel's TrimStrings and ConvertEmptyStringsToNull middleware convert empty strings to null:

$request->validate([
    'title' => ['required', 'unique:posts', 'max:255'],
    'publish_at' => ['nullable', 'date'],
]);

Form Request Validation

For complex scenarios, create a dedicated form request class:

php artisan make:request StorePostRequest

class StorePostRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true;
    }

    public function rules(): array
    {
        return [
            'title' => ['required', 'unique:posts', 'max:255'],
            'body' => ['required'],
        ];
    }
}

Use it directly as a controller method's type-hinted parameter; if validation fails, Laravel handles the redirect/response automatically before the method body ever runs:

public function store(StorePostRequest $request): RedirectResponse
{
    $validated = $request->validated();
    $validated = $request->safe()->only(['name', 'email']);
    return redirect('/posts');
}

Manually Creating Validators

use Illuminate\Support\Facades\Validator;

$validator = Validator::make($request->all(), [
    'title' => ['required', 'unique:posts', 'max:255'],
    'body' => ['required'],
]);

if ($validator->fails()) {
    return redirect('/post/create')
        ->withErrors($validator)
        ->withInput();
}

$validated = $validator->validated();

Working with Error Messages

$errors = $validator->errors();
echo $errors->first('email');
foreach ($errors->get('email') as $message) { /* ... */ }
if ($errors->has('email')) { /* ... */ }

Customizing Error Messages

$validator = Validator::make($input, $rules, $messages = [
    'required' => 'The :attribute field is required.',
    'email.required' => 'We need to know your email address!',
]);

Named Error Bags

For multiple forms on one page, store errors under a named bag:

return redirect('/register')->withErrors($validator, 'login');

Access in a view with {{ $errors->login->first('email') }}.

Performing Additional Validation

$validator->after(function ($validator) {
    if ($this->somethingElseIsInvalid()) {
        $validator->errors()->add('field', 'Something is wrong with this field!');
    }
});

Common Validation Rules

Laravel ships dozens of built-in validation rules covering the vast majority of everyday validation needs. Frequently used ones include:

required — the field must be present and not empty
email — must be a properly formatted email address
numeric / integer — must be a number, or specifically an integer
min:value / max:value — minimum/maximum for strings (length), numbers (value), and arrays (count)
between:min,max — value must fall within the given range
in:foo,bar — value must be one of the given list
exists:table,column — value must exist in the given database table/column
unique:table,column — value must not already exist in the given database table/column
confirmed — expects a matching {field}_confirmation input, commonly used for password confirmation
date / date_format:format — must be a valid date, optionally matching a specific format
array — the field must be a PHP array
image — an uploaded file must be an image (jpg, png, bmp, gif, svg, or webp)

Validating Arrays

When the request contains array input, rules can validate both the array itself and its individual elements using dot notation with a wildcard:

$validator = Validator::make($request->all(), [
    'photos' => 'required|array|max:5',
    'photos.*' => 'image|mimes:jpg,png',
]);

Here, photos must itself be an array with no more than 5 elements, and every individual element of that array must independently satisfy the image and mimes rules.

Validating File Uploads

$request->validate([
    'attachment' => ['required', 'file', 'mimes:pdf,docx', 'max:2048'],
]);

The max rule here is interpreted in kilobytes for file inputs, distinct from how it behaves for strings (character count) or numeric values (numeric maximum).

Conditionally Adding Rules

Use Rule::when or a Validator instance's sometimes method to add rules only when a certain condition is true, which avoids validating fields that aren't even relevant given the rest of the submitted data:

$validator->sometimes('reason', 'required|max:500', function ($input) {
    return $input->games >= 100;
});

Custom Rule Objects

For validation logic that doesn't fit any built-in rule and is reused across multiple forms, create a dedicated rule object:

php artisan make:rule Uppercase

class Uppercase implements ValidationRule
{
    public function validate(string $attribute, mixed $value, Closure $fail): void
    {
        if (strtoupper($value) !== $value) {
            $fail('The :attribute must be uppercase.');
        }
    }
}

Use it inside a rules array just like a built-in rule:

$request->validate([
    'name' => ['required', new Uppercase],
]);
