Effective Go — Concurrency and Language Basics

Introduction

Go is an open-source programming language focusing on simplicity, reliability, and efficiency, designed to make it easy to build software at scale. To write Go well, it's important to understand its properties and idioms.

Formatting

Formatting issues are handled by the `gofmt` program (also available as `go fmt`), which reads Go programs and emits source in a standard style. All Go code in standard packages has been formatted with `gofmt`.

Key formatting principles:
- Indentation: Use tabs; `gofmt` emits them by default
- Line length: No limit; wrap long lines and indent with an extra tab
- Parentheses: Go needs fewer than C and Java; control structures don't require them

Names

Names have semantic effect in Go: visibility outside a package is determined by whether the first character is upper case.

Package Names
Give lower case, single-word names. Avoid underscores or mixedCaps. The package name is the base name of its source directory. Example: `src/encoding/base64` is imported as `"encoding/base64"` but has name `base64`.

Getters
Don't put `Get` into the getter's name. If you have a field `owner` (unexported), the getter should be `Owner` (exported). A setter would be `SetOwner`.

```go
owner := obj.Owner()
if owner != user {
    obj.SetOwner(user)
}
```

Interface Names
One-method interfaces are named by the method name plus an `-er` suffix. Examples: `Reader`, `Writer`, `Formatter`, `CloseNotifier`. Canonical signatures: `Read`, `Write`, `Close`, `Flush`, `String`.

Control Structures

If
```go
if x > 0 {
    return y
}
```
Mandatory braces encourage writing simple if statements on multiple lines. If and switch accept initialization statements:
```go
if err := file.Chmod(0664); err != nil {
    log.Print(err)
    return err
}
```

For
The Go `for` loop unifies C's `for` and `while`:
```go
for init; condition; post { }   // Like a C for
for condition { }               // Like a C while
for { }                         // Like a C for(;;)
```
Range loops manage iteration over arrays, slices, strings, maps, and channels:
```go
for key, value := range oldMap {
    newMap[key] = value
}
```
Use the blank identifier to discard unwanted values:
```go
sum := 0
for _, value := range array {
    sum += value
}
```

Switch
Go's `switch` is more general than C's. Expressions need not be constants or integers; cases are evaluated top to bottom until a match is found.
```go
func unhex(c byte) byte {
    switch {
    case '0' <= c && c <= '9':
        return c - '0'
    case 'a' <= c && c <= 'f':
        return c - 'a' + 10
    case 'A' <= c && c <= 'F':
        return c - 'A' + 10
    }
    return 0
}
```

Type Switch
A type switch discovers the dynamic type of an interface variable:
```go
switch t := t.(type) {
default:
    fmt.Printf("unexpected type %T\n", t)
case bool:
    fmt.Printf("boolean %t\n", t)
case int:
    fmt.Printf("integer %d\n", t)
}
```

Functions

Multiple Return Values
Go functions can return multiple values, useful for returning both a value and an error indication:
```go
func (file *File) Write(b []byte) (n int, err error)
```

Defer
The `defer` statement schedules a function call to be run immediately before the enclosing function returns:
```go
func Contents(filename string) (string, error) {
    f, err := os.Open(filename)
    if err != nil {
        return "", err
    }
    defer f.Close()  // f.Close will run when we're finished.
    // ... read file ...
}
```
Advantages: guarantees you'll never forget to close the file, and the close sits near the open, which is clearer. Deferred functions execute in LIFO order.

Data

Allocation with new vs make
`new(T)` allocates zeroed storage for type T and returns `*T`. The built-in `make(T, args)` creates slices, maps, and channels only, and returns an *initialized* value of type T (not `*T`):
```go
var p *[]int = new([]int)        // *p == nil; rarely useful
var v []int = make([]int, 100)   // v refers to a new array of 100 ints
```

Slices
Slices wrap arrays and provide a more general interface to sequences. Slices hold references to an underlying array — assigning one slice to another means both refer to the same array.

Maps
Maps associate values of one type (key) with values of another (element). Looking up a non-existent key returns the zero value. To distinguish missing entries from zero values, use the "comma ok" idiom:
```go
seconds, ok := timeZone[tz]
```

Interfaces

Interfaces specify behavior. A type implements an interface by implementing its methods:
```go
type Reader interface {
    Read(p []byte) (n int, err error)
}
type Writer interface {
    Write(p []byte) (n int, err error)
}
```
Since almost anything can have methods, almost anything can satisfy an interface.

Concurrency

Share by Communicating

Concurrent programming in Go encourages sharing values via channels rather than sharing memory directly:

> Do not communicate by sharing memory; instead, share memory by communicating.

This approach makes it easier to write clear, correct programs. Go's concurrency originates in Hoare's Communicating Sequential Processes (CSP).

Goroutines

A goroutine is a function executing concurrently with other goroutines in the same address space. It is lightweight — costing little more than stack space allocation — and stacks start small and grow as required.

Goroutines are multiplexed onto multiple OS threads, so if one blocks (e.g., on I/O), others continue to run. This is the key difference from OS threads: goroutines are managed by the Go runtime scheduler (an M:N scheduler mapping many goroutines onto a smaller number of OS threads), whereas threads are managed directly by the operating system and are much heavier — each OS thread has a large, fixed-size stack (often 1-8MB) and costs more to create and context-switch, while a goroutine starts with a small stack (a few KB) that grows and shrinks as needed.

Prefix a function or method call with the `go` keyword to run it in a new goroutine:
```go
go list.Sort()  // run list.Sort concurrently; don't wait for it.
```
Function literals are useful in goroutine invocations:
```go
func Announce(message string, delay time.Duration) {
    go func() {
        time.Sleep(delay)
        fmt.Println(message)
    }()  // Note the parentheses - must call the function.
}
```
Function literals are closures; the implementation ensures variables referred to by the function survive as long as they are active.

Channels

Channels are allocated with `make`. The optional integer parameter sets the buffer size; the default is zero (unbuffered):
```go
ci := make(chan int)            // unbuffered channel of integers
cs := make(chan *os.File, 100)  // buffered channel of pointers
```
Unbuffered channels combine communication with synchronization — guaranteeing two goroutines are in a known state.

Example: waiting for a goroutine to complete:
```go
c := make(chan int)
go func() {
    list.Sort()
    c <- 1  // Send a signal; value does not matter.
}()
doSomethingForAWhile()
<-c   // Wait for sort to finish; discard sent value.
```

How they work:
- Receivers always block until there is data to receive.
- For unbuffered channels, the sender blocks until the receiver has received the value.
- For buffered channels, the sender blocks only until the value has been copied to the buffer.

Buffered channels as semaphores:
```go
var sem = make(chan int, MaxOutstanding)

func handle(r *Request) {
    sem <- 1    // Wait for active queue to drain.
    process(r)  // May take a long time.
    <-sem       // Done; enable next request to run.
}
```

Parallelization

Concurrency is structuring a program as independently executing components. Parallelism is executing calculations in parallel on multiple CPUs for efficiency. Go is a concurrent language, not a parallel one — not all parallelization problems fit Go's model. Goroutines can run in parallel across multiple CPU cores when `GOMAXPROCS` allows it, but concurrency (structure) and parallelism (simultaneous execution) are distinct concepts.

Errors

Library routines should return error indications to the caller. Go's multivalue return makes this easy:
```go
type error interface {
    Error() string
}
```
By convention, errors have type `error`.

Panic and Recover

For unrecoverable errors, use the built-in `panic` function. The built-in `recover` function regains control after a panic, and is only useful inside deferred functions:
```go
func safelyDo(work *Work) {
    defer func() {
        if err := recover(); err != nil {
            log.Println("work failed:", err)
        }
    }()
    do(work)
}
```
In this example, if `do(work)` panics, the result is logged and the goroutine exits cleanly without disturbing others.


Go: Structs and Methods

Source: https://go.dev/tour/methods and https://go.dev/tour/moretypes

Structs

A struct is a collection of fields, used to group related data together into a single named type. Struct fields are accessed using a dot.

type Vertex struct {
    X int
    Y int
}

func main() {
    v := Vertex{1, 2}
    v.X = 4
    fmt.Println(v.X, v.Y)
}

Pointers to Structs

A field of a struct can be accessed through a struct pointer. To access the field X of a struct when we have the struct pointer p we could write (*p).X, but this notation is cumbersome, so the language permits writing p.X instead, without the explicit dereference:

func main() {
    v := Vertex{1, 2}
    p := &v
    p.X = 1e9
    fmt.Println(v)
}

Struct Literals

A struct literal denotes a newly allocated struct value by listing the values of its fields. You can list just a subset of fields by using the Name: syntax, and the order of named fields is irrelevant. The special prefix & returns a pointer to the struct value.

var (
    v1 = Vertex{1, 2}  // has type Vertex
    v2 = Vertex{X: 1}  // Y:0 is implicit
    v3 = Vertex{}      // X:0 and Y:0
    p  = &Vertex{1, 2} // has type *Vertex
)

Methods

Go does not have classes, but you can define methods on types. A method is a function with a special receiver argument, which appears in its own argument list between the func keyword and the method name. The receiver has type Vertex in this example:

type Vertex struct {
    X, Y float64
}

func (v Vertex) Abs() float64 {
    return math.Sqrt(v.X*v.X + v.Y*v.Y)
}

func main() {
    v := Vertex{3, 4}
    fmt.Println(v.Abs())
}

Remember: a method is just a function with a receiver argument. Here Abs is written as a regular function with no change in functionality:

func Abs(v Vertex) float64 {
    return math.Sqrt(v.X*v.X + v.Y*v.Y)
}

You can only declare a method with a receiver whose type is defined in the same package as the method. You cannot declare a method with a receiver whose type is defined in another package, including basic types like int.

Pointer Receivers

You can declare methods with pointer receivers, meaning the receiver type has the literal syntax *T for some type T. The method's receiver, in that case, is a pointer to the value the method was called on, so the method can modify the value that the receiver points to:

func (v *Vertex) Scale(f float64) {
    v.X = v.X * f
    v.Y = v.Y * f
}

func main() {
    v := Vertex{3, 4}
    v.Scale(10)
    fmt.Println(v.Abs())
}

With a value receiver, the method operates on a copy of the original value, so any modification made inside the method has no effect on the original. Methods with pointer receivers can modify the value that the receiver points to, as Scale does above. Since methods are often needed to modify their receiver, pointer receivers are more common than value receivers.

Pointers and Functions

Functions that take a pointer argument must be called with a pointer:

func ScaleFunc(v *Vertex, f float64) {
    v.X = v.X * f
    v.Y = v.Y * f
}

But methods with pointer receivers take either a value or a pointer as the receiver when they are called:

v.Scale(5)  // OK, Go interprets this as (&v).Scale(5) since Scale has a pointer receiver

p := &v
p.Scale(10) // also OK

For the statement v.Scale(5), even though v is a value and not a pointer, the method with the pointer receiver is called automatically. That is, as a convenience, Go interprets the statement v.Scale(5) as (&v).Scale(5) since the Scale method has a pointer receiver.

The equivalent thing happens in the reverse direction: functions that take a value argument must take a value of that specific type, while methods with value receivers take either a value or a pointer as the receiver when they are called.

func (v Vertex) Abs() float64 {
    return math.Sqrt(v.X*v.X + v.Y*v.Y)
}

func main() {
    v := Vertex{3, 4}
    fmt.Println(v.Abs())

    p := &v
    fmt.Println(p.Abs()) // also OK, Go interprets this as (*p).Abs()
}

Choosing a Value or Pointer Receiver

There are two reasons to use a pointer receiver. First, the method can modify the value that its receiver points to. Second, this avoids copying the value on each method call, which can be more efficient if the receiver is a large struct.

In general, all methods on a given type should have either value or pointer receivers, but not a mixture of both — pick whichever is appropriate and be consistent.

Methods and Interfaces

Interfaces are named collections of method signatures. A type implements an interface by implementing its methods; there is no explicit "implements" keyword. If a type has a method with a pointer receiver, only *T (not T) satisfies the interface, since the method set of a pointer type *T includes methods with either receiver type, but the method set of a value type T only includes methods with value receivers.

type Shape interface {
    Area() float64
}

type Rectangle struct {
    Width, Height float64
}

func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

func main() {
    var s Shape = Rectangle{Width: 3, Height: 4}
    fmt.Println(s.Area())
}

Because Rectangle's Area method has a value receiver, both Rectangle and *Rectangle satisfy the Shape interface. If Area had instead been declared with a pointer receiver, func (r *Rectangle) Area() float64, only *Rectangle would satisfy Shape, and passing a plain Rectangle value where a Shape is expected would fail to compile.


Go: Arrays, Slices, and Maps in Depth

Source: https://go.dev/blog/slices-intro and https://go.dev/tour/moretypes

Arrays

An array in Go is a fixed-size sequence of elements of a single type, where the size is part of the type itself. This means [4]int and [5]int are different, incompatible types:

var a [4]int
a[0] = 1
i := a[0]
// i == 1

Arrays are rarely used directly in idiomatic Go code, mostly because their fixed length makes them inflexible; slices are used far more often in practice.

Slices

A slice is a flexible, variable-length view onto the elements of an underlying array. In practice, slices are much more common than arrays in Go code. A slice type is written []T, where T is the element type, with no length specified:

primes := [6]int{2, 3, 5, 7, 11, 13}
var s []int = primes[1:4]
fmt.Println(s) // [3 5 7]

Slices are like references to an underlying array; changing an element of a slice modifies the corresponding element of its underlying array, and other slices sharing the same underlying array observe those changes too.

Slice Length and Capacity

A slice has both a length (the number of elements it currently contains) and a capacity (the number of elements in the underlying array, counting from the first element in the slice). len(s) and cap(s) report these respectively:

s := []int{2, 3, 5, 7, 11, 13}
fmt.Println(len(s), cap(s)) // 6 6

s = s[:0]
fmt.Println(len(s), cap(s)) // 0 6 — capacity unaffected by re-slicing

s = s[:4]
fmt.Println(len(s), cap(s)) // 4 6 — can extend length back up to capacity

You can extend a slice's length as long as there's enough capacity — re-slicing beyond capacity produces a runtime panic.

Creating Slices with make

The built-in make function allocates a zeroed array and returns a slice referring to it, letting you specify length and, optionally, a separate capacity:

a := make([]int, 5)     // len(a)=5, cap(a)=5
b := make([]int, 0, 5)  // len(b)=0, cap(b)=5

Appending to a Slice

The built-in append function adds elements to the end of a slice, growing the underlying array (by allocating a new, larger one and copying the old data over) if the existing array doesn't have enough capacity:

var s []int
s = append(s, 1)
s = append(s, 2, 3, 4)
fmt.Println(s) // [1 2 3 4]

Because append may or may not allocate a new underlying array depending on remaining capacity, it's essential to always assign append's return value back to a variable — the input slice header itself is never modified in place, only potentially the underlying array's contents.

Slices of Slices

Slices can contain any type, including other slices, which is the standard way to represent a 2D grid or matrix in Go:

board := [][]string{
    {"_", "_", "_"},
    {"_", "_", "_"},
    {"_", "_", "_"},
}
board[0][0] = "X"

Range over Slices

The range form of the for loop iterates over a slice, yielding the index and a copy of the element at each step:

for i, v := range pow {
    fmt.Printf("2%d = %d\n", i, v)
}

Skip the index or value with the blank identifier _ if you only need one of them.

Maps

A map maps keys to values. The zero value of a map is nil, and a nil map has no keys and cannot have keys added — it must be initialized with make first:

m := make(map[string]Vertex)
m["Bell Labs"] = Vertex{40.68433, -74.39967}
fmt.Println(m["Bell Labs"])

Map literals are similar to struct literals, but the keys are required:

var m = map[string]Vertex{
    "Bell Labs": {40.68433, -74.39967},
    "Google":    {37.42202, -122.08408},
}

Mutating Maps

Insert or update an element: m[key] = elem
Retrieve an element: elem = m[key]
Delete an element: delete(m, key)
Test whether a key is present, using the "comma ok" idiom: elem, ok = m[key]

If key is not in the map, elem is the zero value for the map's element type, and ok is false — this is the standard way to distinguish "the key exists and maps to the zero value" from "the key doesn't exist at all", which a plain m[key] lookup alone cannot tell apart.

Slices Share Underlying Arrays — A Common Gotcha

Because slices reference an underlying array rather than owning independent storage, two slices derived from the same array (via slicing operations) can silently affect each other:

s1 := []int{1, 2, 3, 4, 5}
s2 := s1[1:3]
s2[0] = 99
fmt.Println(s1) // [1 99 3 4 5] — s1 was modified through s2

This aliasing behavior is a frequent source of subtle bugs, especially when passing slices into functions that might append to or mutate them; when independent copies are genuinely needed, use the built-in copy function to explicitly duplicate the underlying data rather than relying on a fresh slicing operation, which only creates a new slice header, not new backing storage.


Go: Getting Started with Generics

Source: https://go.dev/doc/tutorial/generics

Overview

Generics let you declare and use functions or types that work with any of a set of types provided by the calling code, without duplicating logic for every concrete type. This guide walks through the progression from writing type-specific functions to writing a single generic function, then to declaring a reusable type constraint.

The Problem Being Solved

Consider two maps of different value types, one holding int64 values and one holding float64 values. Without generics, summing the values in each map requires writing near-identical functions, one per type.

Non-Generic Functions

Two separate functions, each summing a map of a specific value type:

func SumInts(m map[string]int64) int64 {
    var s int64
    for _, v := range m {
        s += v
    }
    return s
}

func SumFloats(m map[string]float64) float64 {
    var s float64
    for _, v := range m {
        s += v
    }
    return s
}

Calling them:

func main() {
    ints := map[string]int64{
        "first":  34,
        "second": 12,
    }

    floats := map[string]float64{
        "first":  35.98,
        "second": 26.99,
    }

    fmt.Printf("Non-Generic Sums: %v and %v\n",
        SumInts(ints),
        SumFloats(floats))
}

Output: Non-Generic Sums: 46 and 62.97

Adding a Generic Function

Instead of two separate functions, you can write one generic function using type parameters declared in square brackets:

func SumIntsOrFloats[K comparable, V int64 | float64](m map[K]V) V {
    var s V
    for _, v := range m {
        s += v
    }
    return s
}

Here K and V are type parameters. K comparable means K can be any type whose values support the == and != operators, which is required because K is used as a map key. V int64 | float64 is a union constraint that allows the value type to be either int64 or float64.

Calling with explicit type arguments:

fmt.Printf("Generic Sums: %v and %v\n",
    SumIntsOrFloats[string, int64](ints),
    SumIntsOrFloats[string, float64](floats))

Calling with inferred type arguments — in most cases you can omit the type arguments entirely and let the Go compiler infer them from the function's regular arguments:

fmt.Printf("Generic Sums, type parameters inferred: %v and %v\n",
    SumIntsOrFloats(ints),
    SumIntsOrFloats(floats))

Declaring a Type Constraint

When you want to reuse a constraint across multiple functions, declare it as an interface:

type Number interface {
    int64 | float64
}

Then reference it in a generic function's type parameter list:

func SumNumbers[K comparable, V Number](m map[K]V) V {
    var s V
    for _, v := range m {
        s += v
    }
    return s
}

Calling it works the same way as before:

fmt.Printf("Generic Sums with Constraint: %v and %v\n",
    SumNumbers(ints),
    SumNumbers(floats))

Complete Example

package main

import "fmt"

type Number interface {
    int64 | float64
}

func main() {
    ints := map[string]int64{
        "first": 34,
        "second": 12,
    }

    floats := map[string]float64{
        "first": 35.98,
        "second": 26.99,
    }

    fmt.Printf("Non-Generic Sums: %v and %v\n",
        SumInts(ints),
        SumFloats(floats))

    fmt.Printf("Generic Sums: %v and %v\n",
        SumIntsOrFloats[string, int64](ints),
        SumIntsOrFloats[string, float64](floats))

    fmt.Printf("Generic Sums, type parameters inferred: %v and %v\n",
        SumIntsOrFloats(ints),
        SumIntsOrFloats(floats))

    fmt.Printf("Generic Sums with Constraint: %v and %v\n",
        SumNumbers(ints),
        SumNumbers(floats))
}

func SumInts(m map[string]int64) int64 {
    var s int64
    for _, v := range m {
        s += v
    }
    return s
}

func SumFloats(m map[string]float64) float64 {
    var s float64
    for _, v := range m {
        s += v
    }
    return s
}

func SumIntsOrFloats[K comparable, V int64 | float64](m map[K]V) V {
    var s V
    for _, v := range m {
        s += v
    }
    return s
}

func SumNumbers[K comparable, V Number](m map[K]V) V {
    var s V
    for _, v := range m {
        s += v
    }
    return s
}

Why This Matters

Generics reduce code duplication while preserving type safety at compile time. Before generics existed in Go, developers often resorted to the empty interface (interface{}) plus runtime type assertions to write reusable containers and algorithms, which pushed type errors from compile time to runtime. Type parameters let the compiler check usage at compile time instead, catching mismatched types before the program ever runs, while still allowing a single function body to serve many concrete types.

Union constraints like int64 | float64 restrict which underlying types are permitted, and comparable is a predeclared constraint satisfied by any type whose values can be compared with == and !=, which is exactly what's needed for map keys. These constraints give the compiler enough information to type-check operations used inside the generic function, such as the += operator in SumNumbers, without allowing arbitrary unrelated types to be substituted.
