Request Lifecycle - How Requests Flow Through Web Frameworks

Introduction

The request lifecycle is the journey of an HTTP request from arrival at a web server through processing and response generation. Understanding this flow is essential for effective web development, debugging, and optimization.

Different frameworks organize this flow differently, but the fundamental stages are similar across Go, PHP, Python, and Java frameworks.

Generic Request Lifecycle Stages

1. Server Receives Request
   - TCP connection established
   - HTTP request parsed
   - Request line (GET /api/users HTTP/1.1)
   - Headers extracted
   - Body read (if POST/PUT/PATCH)

2. Framework Initialization
   - Framework boots up
   - Configuration loaded
   - Dependencies instantiated
   - Service container/IoC initialized

3. Routing
   - URL matched against routes
   - Route handler determined
   - Parameters extracted

4. Middleware Processing (Request Phase)
   - Global middleware executes
   - Route-specific middleware executes
   - Authentication, CORS, compression, etc.
   - Request object modified with context
   - Middleware can short-circuit with response

5. Request Dispatching
   - Controller/Handler method called
   - Request object passed as parameter
   - Route parameters injected

6. Business Logic Execution
   - Controller coordinates Model and View
   - Service/business logic methods called
   - Database queries executed
   - Data processing happens

7. Response Generation
   - Data prepared for response
   - View rendering
   - Serialization (JSON, HTML, etc.)
   - Response object created

8. Middleware Processing (Response Phase)
   - Response middleware executes in reverse order
   - Response headers modified
   - Body compression
   - Logging

9. Sending Response
   - Status code set
   - Headers sent to client
   - Body sent to client

10. Connection Cleanup
    - Resources freed
    - Database connections returned to pool
    - Request context destroyed

Lifecycle in Go with Gin Framework

Gin follows a clean, explicit request flow:

```go
package main

import "github.com/gin-gonic/gin"

func main() {
  router := gin.Default()

  // Global middleware - runs for every request
  router.Use(gin.Logger())      // Logging
  router.Use(gin.Recovery())    // Error recovery

  // Route-specific middleware and handler
  router.POST("/users", authMiddleware(), createUserHandler)
}

func authMiddleware() gin.HandlerFunc {
  return func(c *gin.Context) {
    token := c.GetHeader("Authorization")
    if token == "" {
      c.JSON(401, gin.H{"error": "No auth token"})
      c.Abort()  // Stop middleware chain
      return
    }

    // Attach to context for later use
    c.Set("user_id", 123)

    c.Next()  // Continue to next middleware/handler
  }
}

func createUserHandler(c *gin.Context) {
  // Extract user_id from context (added by middleware)
  userID, _ := c.Get("user_id")

  // Parse request body
  var req CreateUserRequest
  if err := c.BindJSON(&req); err != nil {
    c.JSON(400, gin.H{"error": "Invalid request"})
    return
  }

  // Call service layer
  user, err := userService.Create(req.Name, req.Email)
  if err != nil {
    c.JSON(500, gin.H{"error": "Failed to create user"})
    return
  }

  // Send response
  c.JSON(201, user)
}
```

Flow:
1. Request arrives
2. Router matches POST /users
3. authMiddleware runs
   - Checks authentication
   - Attaches user_id to context
   - Calls c.Next()
4. createUserHandler runs
   - Accesses user_id from context
   - Validates request body
   - Calls service layer
   - Sends response

Lifecycle in PHP with Laravel Framework

Laravel has more automatic, convention-based flow:

```php
<?php

Route::post('/users', [UserController::class, 'store'])->middleware('auth');

class UserController {
  public function __construct(UserService $service) {
    $this->service = $service;  // Dependency injection
  }

  public function store(Request $request) {
    // Middleware already ran (auth checked)

    // Validate input (automatic)
    $validated = $request->validate([
      'name' => 'required|string',
      'email' => 'required|email'
    ]);

    // Call service layer
    $user = $this->service->create($validated);

    // Return response
    return response()->json($user, 201);
  }
}
```

Laravel automatically:
1. Loads service providers
2. Routes request to controller method
3. Runs middleware
4. Injects dependencies
5. Calls method
6. Sends response

Lifecycle in Python with Django Framework

Django follows URL routing to views:

```python
# urls.py
from django.urls import path
from . import views

urlpatterns = [
  path('api/users/', views.create_user, name='create_user'),
]

# views.py
from django.http import JsonResponse
from django.views.decorators.http import require_http_methods
from django.views.decorators.csrf import csrf_exempt

@require_http_methods(["POST"])
@csrf_exempt
def create_user(request):
  # Middleware already processed request

  # Parse JSON body
  import json
  try:
    data = json.loads(request.body)
  except json.JSONDecodeError:
    return JsonResponse({'error': 'Invalid JSON'}, status=400)

  # Validate
  if 'name' not in data or 'email' not in data:
    return JsonResponse({'error': 'Missing fields'}, status=400)

  # Call model/service
  user = User.objects.create(name=data['name'], email=data['email'])

  # Return response
  return JsonResponse({'id': user.id, 'name': user.name}, status=201)
```

Django processes:
1. Loads middleware
2. Routes to view function
3. Middleware processes request
4. View executes
5. Response generated
6. Middleware processes response

Lifecycle in Java with Spring Boot

Spring Boot uses annotations and convention:

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

  private UserService userService;

  // Dependency injection via constructor
  public UserController(UserService userService) {
    this.userService = userService;
  }

  @PostMapping
  @PreAuthorize("hasRole('USER')")  // Annotation-based auth
  public ResponseEntity<User> createUser(@RequestBody CreateUserRequest req) {
    // Validation already run by Spring
    // Authentication/Authorization already checked

    // Call service
    User user = userService.create(req.getName(), req.getEmail());

    // Return response with status code
    return ResponseEntity.status(201).body(user);
  }
}
```

Spring processes:
1. Loads application context
2. Registers beans from @Configuration
3. Routes request to controller method
4. Applies interceptors/filters
5. Checks @PreAuthorize annotations
6. Validates @RequestBody
7. Calls method with injected dependencies
8. Serializes response to JSON
9. Sends response with status code

Key Concepts in Request Lifecycle

Middleware Chain

Middleware executes in order for request, then in reverse order for response:

```
Request → MW1 → MW2 → MW3 → Handler → MW3 → MW2 → MW1 → Response
         Request       Response
         Phase         Phase
```

Middleware can:
- Examine request before handler
- Modify request object
- Short-circuit and send response
- Process response after handler
- Modify response headers/body

Context/Request Object

Request object carries data through the lifecycle:
- HTTP method, path, headers, body
- Route parameters
- Query parameters
- Authenticated user
- Custom data added by middleware

Different frameworks represent this differently (gin.Context, Django request, Spring RequestContext).

Dependency Injection

Modern frameworks use DI to provide services to handlers:

```java
// Spring Boot injects UserService
public UserController(UserService userService) {
  this.userService = userService;
}

// Container creates instances and passes them
```

This enables loose coupling and testability.

Error Handling in Request Lifecycle

Errors at different stages:

Routing stage: No matching route → 404 Not Found

Middleware stage: Authentication fails → 401 Unauthorized, request stops

Handler stage: Business logic error → appropriate status code (400, 500, etc)

Response stage: Error during serialization → 500 Internal Server Error

Most frameworks have global error handlers:

```java
@ControllerAdvice
public class GlobalExceptionHandler {
  @ExceptionHandler(NotFoundException.class)
  public ResponseEntity<?> handleNotFound(NotFoundException e) {
    return ResponseEntity.status(404).body(new Error("Not found"));
  }
}
```

Request Lifecycle Optimization

Database Connection Pooling: Reuse connections across requests

Caching: Cache responses to avoid recomputation

Lazy Loading: Load data only when needed

Asynchronous Processing: Offload slow operations to background jobs

Load Balancing: Distribute requests across servers

Monitoring: Track slow requests and optimize

Common Lifecycle Patterns

1. Request Validation Pattern
   - Validate input early
   - Return error immediately if invalid
   - Avoid processing invalid data

2. Authorization Pattern
   - Check permissions early
   - Return 403 if unauthorized
   - Prevents wasting resources on unauthorized requests

3. Database Transaction Pattern
   - Begin transaction
   - Execute queries
   - Commit or rollback
   - Often handled automatically by ORM

4. Logging Pattern
   - Log request received
   - Log processing steps
   - Log response sent
   - Help with debugging and monitoring

5. Error Response Pattern
   - Catch exceptions
   - Log error
   - Return appropriate status code
   - Return meaningful error message

Understanding the request lifecycle helps you:
- Debug issues effectively
- Optimize performance
- Implement features correctly
- Understand middleware behavior
- Write better error handling

The lifecycle may look different across frameworks, but the core concepts remain consistent.
