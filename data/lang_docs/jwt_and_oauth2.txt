JWT and OAuth2 - Token-Based Authentication and Delegation

Introduction

JWT (JSON Web Tokens) and OAuth2 are complementary technologies for modern authentication and authorization. JWT is a token format for encoding claims, while OAuth2 is a framework for delegating access. Together, they enable secure, scalable authentication where clients use tokens instead of repeatedly transmitting credentials.

Understanding both is essential for building modern web applications and APIs.

JSON Web Tokens (JWT)

JWT is a compact, URL-safe way to represent claims to be transferred between parties. A claim is a piece of information about an entity (typically a user). JWTs are digitally signed, ensuring the claims cannot be modified without detection.

JWT Structure

A JWT consists of three Base64URL-encoded parts separated by periods:

header.payload.signature

Header (JOSE Header): Contains metadata about the token, including the signing algorithm and token type.

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

Payload (Claims Set): Contains the actual claims, which are statements about the entity and any additional metadata.

```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "email": "john@example.com",
  "iat": 1516239022,
  "exp": 1516242622
}
```

Signature: Generated by signing the header and payload with a secret key or private key. Ensures the token hasn't been tampered with.

```
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret
)
```

Example JWT:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Registered Claims

JWT defines seven standard claims that have specific meanings:

iss (Issuer): Principal that issued the token (e.g., "auth.example.com")
sub (Subject): Principal the token describes, typically user ID (e.g., "user123")
aud (Audience): Intended recipients of the token (e.g., ["api.example.com"])
exp (Expiration Time): Unix timestamp when the token expires
nbf (Not Before): Unix timestamp before which the token is not valid
iat (Issued At): Unix timestamp when the token was created
jti (JWT ID): Unique identifier for the token, prevents replay attacks

Example with registered claims:

```json
{
  "iss": "https://auth.example.com",
  "sub": "user123",
  "aud": "api.example.com",
  "exp": 1626184200,
  "iat": 1626180600,
  "nbf": 1626180600,
  "jti": "unique-token-id"
}
```

Custom Claims

Beyond registered claims, you can include custom claims for application-specific data:

```json
{
  "sub": "user123",
  "name": "John Doe",
  "role": "admin",
  "permissions": ["read", "write", "delete"],
  "organization_id": "org456"
}
```

Be cautious with sensitive data in JWTs - the payload is only Base64-encoded, not encrypted. Anyone can decode it.

JWT Algorithms

JWTs can be signed or encrypted. Signing provides integrity (can't be modified), while encryption provides confidentiality (can't be read).

Signing algorithms:

HS256 (HMAC SHA-256): Symmetric algorithm using a shared secret. Server signs and verifies with the same secret. Fastest, suitable for single-server systems.

RS256 (RSA SHA-256): Asymmetric algorithm using public/private key pair. Server signs with private key, clients verify with public key. Suitable for distributed systems where clients shouldn't have the signing key.

ES256 (ECDSA SHA-256): Elliptic Curve variant. Shorter signatures than RSA but provides same security.

Encryption algorithms: JWE (JSON Web Encryption) provides confidentiality. Less commonly used than signing because the payload is usually not sensitive.

JWT Workflow

1. User authenticates with username and password
2. Server validates credentials
3. Server creates JWT with claims about the user
4. Server signs JWT with secret key
5. Server returns JWT to client
6. Client stores JWT (in localStorage, sessionStorage, or cookie)
7. Client includes JWT in subsequent requests:
   Authorization: Bearer eyJhbGciOiJIUzI1NiI...
8. Server receives request, extracts JWT from Authorization header
9. Server verifies JWT signature using the secret key
10. Server checks exp claim to ensure not expired
11. Server extracts claims from JWT (user ID, role, etc.)
12. Server processes request with JWT claims

JWT Best Practices

Use HTTPS for all JWT transmission - prevents interception
Keep tokens short-lived - exp claim usually 15 minutes to 1 hour
Use refresh tokens for long sessions - short-lived access token + long-lived refresh token
Include only necessary claims - don't bloat the token
Use appropriate signing algorithm - HS256 for single server, RS256 for distributed
Validate exp claim - reject expired tokens
Validate signature on every request - never trust tokens
Store tokens securely - HttpOnly cookies, secure localStorage
Implement token rotation - refresh tokens periodically
Never store sensitive data in JWT payload - it's readable without the secret

JWT Limitations

Tokens can't be revoked immediately - they're valid until expiration
Large tokens increase bandwidth - payload bloat
Signature verification has overhead - added latency
Clock skew issues - token created/verified on different servers with different times

OAuth2

OAuth2 is an authorization framework that enables applications to obtain limited access to user accounts on an HTTP service. It's widely used for delegated authentication (letting Google or GitHub handle login).

Key Concepts

Resource Owner: The user who owns the data/account
Client Application: The application requesting access (e.g., your web app)
Authorization Server: Service that authenticates users and issues tokens (e.g., Google)
Resource Server: API that holds the protected resources
Scopes: Permissions the client is requesting (e.g., read:email, write:posts)

OAuth2 Grant Types

Authorization Code Grant (most common for web apps):

1. User clicks "Login with Google"
2. App redirects to Google's authorization endpoint with client_id, redirect_uri, scope
3. User sees Google login page, logs in, and authorizes the app
4. Google redirects to app's redirect_uri with authorization code
5. App backend exchanges authorization code for access token (secret exchange)
6. App uses access token to request user info from Google
7. App creates local session for user

```
GET https://accounts.google.com/o/oauth2/v2/auth?
  client_id=CLIENT_ID&
  redirect_uri=REDIRECT_URI&
  scope=openid%20email%20profile&
  response_type=code
```

Client Credentials Grant (server-to-server):

1. Server A has client_id and client_secret
2. Server A requests token from authorization server with credentials
3. Authorization server returns access token
4. Server A uses token to call Server B's API

Used for backend services without user interaction.

Authorization Code with PKCE (mobile/SPA apps):

Like Authorization Code Grant but uses Proof Key for Code Exchange for additional security. Essential for mobile and single-page applications where storing client_secret is risky.

Implicit Grant (deprecated): Access token returned directly without authorization code. Was common for SPAs but now deprecated due to security concerns.

Resource Owner Password Credentials (deprecated): User gives username/password to app, app exchanges for token. Deprecated because requires trusting the app with credentials.

Device Flow: For devices without browsers (smart TVs, IoT devices). User authorizes on another device.

OAuth2 Token Types

Access Token: Short-lived token for accessing protected resources. Usually in JWT format.

Refresh Token: Long-lived token used to get new access tokens. Sent to authorization server without user interaction.

ID Token: OpenID Connect token (JWT) containing user identity information. Includes claims about authentication.

Combining OAuth2 with JWT

Modern OAuth2 implementations use JWT for access tokens. When server issues an access token, it's a signed JWT containing:
- User ID
- Issued time (iat)
- Expiration time (exp)
- Scopes/permissions
- Other metadata

Client stores JWT and uses it for subsequent API calls. Server verifies JWT signature and expiration without calling the authorization server again.

OAuth2 Security Considerations

Always use HTTPS - prevents token interception
Validate authorization code once - prevents replay attacks
Keep access tokens short-lived - reduces impact of token compromise
Use PKCE for mobile/SPA apps - protects against authorization code interception
Validate redirect_uri - prevents malicious redirects
Use strong client_secret - long random string
Implement state parameter - prevents CSRF attacks
Validate access token signature - ensures token authenticity

Combining JWT and OAuth2

Modern web applications often use OAuth2 for authentication and JWT for authorization:

1. User logs in via OAuth2 (Google, GitHub, etc.)
2. Authorization server issues JWT containing user info
3. Frontend stores JWT
4. Frontend sends JWT in subsequent API requests
5. Backend validates JWT signature and extracts claims
6. Backend checks exp and validates authorization

This approach combines the delegated authentication of OAuth2 with the simplicity and scalability of JWT tokens.

Practical Examples

User logs in with password, receives JWT:

```
POST /auth/login
Content-Type: application/json

{
  "email": "user@example.com",
  "password": "secure-password"
}

Response 200:
{
  "access_token": "eyJhbGciOiJIUzI1NiI...",
  "refresh_token": "eyJhbGciOiJIUzI1NiI...",
  "expires_in": 3600
}
```

Client uses JWT to access API:

```
GET /api/users/me
Authorization: Bearer eyJhbGciOiJIUzI1NiI...

Response 200:
{
  "id": "user123",
  "name": "John Doe",
  "email": "john@example.com"
}
```

Client refreshes access token:

```
POST /auth/refresh
Content-Type: application/json

{
  "refresh_token": "eyJhbGciOiJIUzI1NiI..."
}

Response 200:
{
  "access_token": "eyJhbGciOiJIUzI1NiI...",
  "expires_in": 3600
}
```

JWT and OAuth2 are foundational technologies for modern authentication and authorization. JWT provides a compact token format, while OAuth2 provides a framework for delegating authentication and authorization.
